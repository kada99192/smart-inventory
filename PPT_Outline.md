### Slide 6 — 核心技術 1：YOLOv11 物件偵測

**標題**
YOLOv11：毫秒級物品定位

**左側：技術說明**

YOLOv11 是 Ultralytics 最新一代即時目標偵測模型，相較 YOLOv8 推理速度提升約 20%，在保持高精度的同時大幅降低計算開銷。

本系統採用輕量版 `yolo11n.pt`，可辨識 COCO 資料集 80 類常見物品，涵蓋瓶裝飲料、蔬果、罐頭等家庭常見食品。

**右側：關鍵參數**

```python
results = model.predict(
    source  = img_bgr,
    conf    = 0.40,      # 信心閾值（可調）
    imgsz   = 640,       # 標準輸入解析度
    verbose = False,
)
# 每個 BBox 自動裁切為 ROI 子圖
# 加入 12px padding 確保文字不被截斷
```

**亮點說明**
信心閾值低於 0.40 的偵測結果將退回以整張圖片作為辨識範圍，確保漏偵測時仍能執行 OCR。

### Slide 7 — 核心技術 2：EasyOCR 多格式日期辨識

**標題**
EasyOCR：讀懂 7 種日期格式

**痛點說明**
食品包裝的到期日印刷格式高度不統一，同一類商品可能出現截然不同的標示方式。

**支援格式一覽**

| 格式 | 範例 | 場景 |
|------|------|------|
| `YYYY/MM/DD` | 2026/03/15 | 台灣本土食品 |
| `YYYY-MM-DD` | 2026-03-15 | 國際標準 |
| `YYYYMMDD` | 20260315 | 雷射印字 |
| `YYYY年MM月DD日` | 2026年03月15日 | 傳統中文標示 |
| `MM/DD/YYYY` | 03/15/2026 | 進口美系商品 |
| `DD.MM.YYYY` | 15.03.2026 | 進口歐系商品 |
| `YY/MM/DD` | 26/03/15 | 簡短日期格式 |

**雙層 OCR 策略**
ROI 子圖 OCR 結果優先；若 ROI 未找到日期，自動 fallback 至全圖 OCR，解決到期日印刷在標籤邊緣的問題。

### Slide 8 — 核心技術 3：智慧去重機制

**標題**
二階去重：讓庫存資料保持乾淨

**流程圖（文字版）**

```
新物品 + 到期日
       │
       ▼
  ┌─────────────┐
  │  精確比對    │ → item_name 完全相同
  │  (SQL)      │   AND expiry_date 完全相同
  └──────┬──────┘
    命中 ↓ 未命中
    SKIP  │
          ▼
  ┌─────────────┐
  │  模糊比對    │ → fuzz.ratio(新名稱, 既有名稱)
  │ (rapidfuzz) │   相同到期日桶內搜尋
  └──────┬──────┘
    ≥85% ↓ <85%
    SKIP  │
          ▼
        INSERT
```

**設計理由**
`fuzz.ratio` 對短字串（< 6 字元）的長度差異敏感，因此閾值設為 85%，兼顧「蘋果汁（大）」≈「蘋果汁（小）」的正確命中，同時避免「蘋果」≠「蘋果醋」的誤判。

**標題**
視覺設計：讓危機感一眼可見

**設計原則**

### Slide 12 — Agent 關鍵決策 1：OCR 辨識邏輯優化

**標題**
Agent 的第一個關鍵決策：雙層 OCR 策略

**問題情境**
單純對 YOLO 裁切出的 ROI 子圖做 OCR，若到期日印刷在 BBox 邊緣外側，會導致完全讀取不到日期。

**Agent 分析過程**
1. 識別到「到期日常印刷於瓶蓋邊緣、封口處，不在主體 BBox 內」
2. 提出「ROI + 全圖並行 OCR，ROI 優先，全圖 fallback」策略
3. 同時針對圖片進行灰階化、上放大、Otsu 二值化的前處理鏈

**解決方案**
```
全圖 OCR ←─────────────── fallback
   ↑
   │ 未找到日期
   │
ROI OCR ──→ 找到日期 ──→ 採用 ROI 結果（更精確）
```

**效益**
日期辨識覆蓋率從僅 1 種格式提升至 7 種格式，並解決邊緣印刷的漏判問題。

### Slide 13 — Agent 關鍵決策 2：去重閾值分析

**標題**
Agent 的第二個關鍵決策：數學驗證去重閾值

**問題情境**
測試腳本 T5 出現 2 個失敗，「可口可樂(罐)」與「可口可樂」的模糊比對未能命中 85% 閾值。

**Agent 自主除錯過程**

Step 1 — 執行測試，觀察失敗訊息
```
✗ 模糊：可口可樂(罐) ≈ 可口可樂
✗ 模糊：牛奶(全脂) ≈ 牛奶
```

Step 2 — 計算實際相似度
```python
fuzz.ratio("可口可樂(罐)", "可口可樂") = 72.7%  # < 85% 閾值
fuzz.ratio("牛奶(全脂)",   "牛奶")     = 50.0%  # < 85% 閾值
```

Step 3 — 根因分析
> `fuzz.ratio = 2M / (len1 + len2)`，公式對長度差異高度敏感。
> 短字串加入後綴（罐、全脂）後，分母增加但分子不足，無法達標。
> 這是**程式正確行為**，是**測試字串選擇錯誤**。

Step 4 — 修正測試，補充反向驗證
```python
# 修正：改用真正相似的長字串（7字元，1字元差異 = 85.7%）
fuzz.ratio("全脂牛奶一公升", "全脂牛奶半公升") = 85.7% ✓
# 補充：短字串低於閾值屬正確行為（反向測試）
assert not db_is_duplicate("可口可樂(罐)", thresh=85)  # 正確拒絕
```

**結果**：從 119/121 → **122/122 全數通過**

### Slide 14 — Agent 關鍵決策 3：CSS 動畫分層設計

**標題**
Agent 的第三個關鍵決策：避免動畫衝突的時序控制

**技術挑戰**
同時觸發 `slideUp`（入場）和 `pulseRed`（脈動）兩個 CSS 動畫時，若時序重疊會導致視覺跳動。

**Agent 設計方案**

```css
.pulse-red {
    animation:
        slideUp  .42s cubic-bezier(.22,.61,.36,1) both,
        /*       ↑ 先完成入場（0.42 秒）                */
        pulseRed 1.9s ease-in-out 0.5s infinite;
        /*                        ↑ delay 0.5s，確保入場結束後才開始脈動 */
}
```

**設計邏輯**
- `slideUp`：0 → 0.42s，完成入場
- 靜止期：0.42s → 0.5s（約 80ms 緩衝）
- `pulseRed`：0.5s 開始，無限循環

**視覺效果**：卡片流暢滑入 → 穩定後開始發出紅色脈動光暈，無任何抖動。

從圖片輸入到庫存警示，整個流程零人工介入。AI Agent 協助在 3 小時內整合 YOLOv11、EasyOCR、SQLite、Streamlit 四大技術棧。

**2. Agent 不只寫程式，更會除錯**
測試失敗時，Agent 自主計算 `fuzz.ratio` 實際值，識別根因為「測試字串選擇錯誤」而非程式 Bug，並補充反向驗證案例，展現超越「程式碼生成」的工程判斷力。

**3. 品質與速度可以兼得**
122 個測試案例、7 種日期格式支援、雙層 OCR 策略、CSS 動畫時序控制——這些工程品質決策，在傳統開發中往往是時間壓力下最先被犧牲的部分。

**結語**
> 當 AI Agent 成為開發夥伴，工程師的價值從「如何寫程式碼」轉移至「如何提出正確的問題」。
